# WebOS 系统架构文档

## 1. 整体分层架构

WebOS 是一个完全运行在浏览器中的 WebAssembly 操作系统，采用经典的多层架构设计，从底层浏览器 API 到上层用户应用程序共分为七个层级。每一层都有明确的职责边界，层与层之间通过定义良好的接口进行通信。这种分层设计确保了系统的模块化、可维护性和可扩展性，使得每一层都可以独立开发和测试，同时也方便未来的功能扩展和性能优化。

```

│                      应用层 (Applications)                       │
│                    .wex 可执行模块 (C++)                          │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────────────┐  │
```

### 各层职责说明

**浏览器 API 层**是最底层，提供 WebAssembly 运行时环境、图形渲染能力（Canvas2D、WebGL、WebGPU）、网络访问能力（Fetch API）、以及持久化存储能力（IndexedDB）。WebOS 不直接与操作系统硬件交互，而是通过浏览器提供的标准 Web API 实现所有底层功能。这一设计决策使得 WebOS 具有天然的可移植性，任何支持 WebAssembly 的现代浏览器都可以作为运行平台。

**宿主运行时层**由 TypeScript 编写，是整个 WebOS 与浏览器交互的桥梁。它包含两个核心组件：DynamicLoader（动态加载器）负责加载和实例化 WASM 模块，读取模块的自定义段信息以确定模块类型；RuntimeBridge（运行时桥接器）负责实现系统调用的宿主端处理逻辑，将 WASM 模块的 syscall 请求转发到对应的浏览器 API 调用。宿主运行时还管理共享的 WebAssembly.Memory 实例，所有模块共享同一块线性内存空间。

**引导加载层**是系统启动的第一个 C 语言组件，由宿主运行时加载执行。它的主要职责包括初始化 GPU 渲染环境、加载内核 WASM 模块、将控制权转交给内核。引导加载层通过 `load_module()` 函数实现模块的按需加载，通过 `boot_log()` 函数输出启动日志信息。

**内核层**是 WebOS 的核心，采用纯 C 语言编写，编译为独立的 WASM 模块。内核在启动时依次初始化内存管理、进程管理、调度器、IPC 通信、动态链接器和系统调用分发表等子系统。内核不直接提供高级服务，而是通过系统调用接口为上层提供服务能力，通过动态链接器加载驱动和服务模块。

**驱动层**将硬件抽象为统一接口。由于 WebOS 运行在浏览器中，驱动层实际上是对浏览器 API 的封装：GPU 驱动封装了 Canvas2D/WebGL/WebGPU 渲染能力，输入驱动封装了键盘和鼠标事件，存储驱动封装了 IndexedDB 持久化存储，网络驱动封装了 Fetch API 网络请求。所有驱动均编译为 `.Wdll` 动态链接库格式，由内核的动态链接器在启动时加载。

**服务层**构建在驱动层之上，提供面向应用的高级服务接口。窗口管理器服务使用 GPU 驱动实现窗口合成和渲染，文件系统服务使用存储驱动实现虚拟文件系统，网络服务使用网络驱动实现 HTTP 请求，应用商店服务提供应用发现和安装能力。服务同样编译为 `.Wdll` 格式，通过 IPC 与应用程序通信。

**应用层**是用户直接交互的最顶层，采用 C++ 编写，编译为 `.wex` 可执行模块。每个应用通过系统调用和 IPC 请求服务层的功能，通过 WM 客户端库创建和管理窗口。

## 2. 模块类型与加载流程

WebOS 定义了三种核心模块类型，每种类型对应不同的文件后缀名、WASM 自定义段标识和运行时行为。模块类型的识别不依赖文件后缀名，而是通过 WASM 二进制文件中的 `module_type` 自定义段实现。这种设计确保了即使文件被重命名，运行时仍然可以正确识别模块类型。

### 模块类型定义表

| 类型 | 文件后缀 | 自定义段值 | 语言 | 描述 |
|------|---------|-----------|------|------|
| 可执行模块 | `.wex` | `module_type="wex"` | C++ | 独立应用程序，导出 `main()` 入口函数 |
| 动态链接库 | `.Wdll` | `module_type="Wdll"` | C | 共享库、驱动或服务，导出公共 API 函数 |
| 库接口描述 | `.wli` | `module_type="wli"` | JSON | 符号描述文件，用于动态链接器的符号解析 |

### 加载流程

```
                    模块加载流程
                    ═══════════

  ┌─────────┐     ┌──────────────┐     ┌──────────────┐
  │ 请求加载 │────▶│  获取 WASM   │────▶│  读取自定义段 │
  │ 模块名称 │     │  二进制数据   │     │  识别类型     │
  └─────────┘     └──────────────┘     └──────┬───────┘
                                              │
                          ┌───────────────────┼───────────────────┐
                          │                   │                   │
                          ▼                   ▼                   ▼
                   ┌──────────┐        ┌──────────┐        ┌──────────┐
                   │  .wex    │        │  .Wdll   │        │  .wli    │
                   │ 可执行   │        │  动态库  │        │  接口描述 │
                   └────┬─────┘        └────┬─────┘        └────┬─────┘
                        │                   │                   │
                        ▼                   ▼                   ▼
                 ┌────────────┐      ┌────────────┐      ┌────────────┐
                 │ 编译 WASM  │      │ 编译 WASM  │      │ 解析 JSON  │
                 │ 实例化     │      │ 实例化     │      │ 注册接口   │
                 └────┬───────┘      │ SIDE_MODULE│      └────────────┘
                      │              └────┬───────┘
                      ▼                   ▼
                 ┌────────────┐      ┌────────────┐
                 │ 创建进程   │      │ 注册导出   │
                 │ 加入调度   │      │ 符号表     │
                 │ 执行 main()│      │ 增加引用   │
                 └────────────┘      └────────────┘
```

**.wex 可执行模块**的加载流程最为完整：宿主运行时的 DynamicLoader 首先通过 Fetch API 获取 WASM 二进制数据，然后读取 `module_type` 自定义段确认类型为 `wex`。接着使用 `WebAssembly.compile()` 编译模块字节码，再通过 `WebAssembly.instantiate()` 实例化模块，传入 `webos.syscall_invoke` 导入函数和共享的 `env.memory`。实例化成功后，内核为该模块创建新的进程控制块（PCB），分配进程 ID，设置进程状态为 `PROC_STATE_READY`，并将其加入调度器的就绪队列。当调度器选中该进程时，调用其导出的 `main()` 函数开始执行。

**.Wdll 动态链接库**的加载流程与可执行模块类似，但使用 Emscripten 的 SIDE_MODULE 模式编译，这意味着它不能独立运行，必须被主模块链接使用。加载时，DynamicLoader 同样识别自定义段确认类型为 `Wdll`，然后编译并实例化模块。实例化后，动态链接器将模块的所有导出函数注册到全局符号表中，以便其他模块通过 `dynlink_resolve()` 查找和调用。每个动态链接库维护一个引用计数，当引用计数降为零时，动态链接器可以卸载该模块释放资源。

**.wli 库接口描述文件**是一个 JSON 格式的文本文件，描述了对应 `.Wdll` 模块的导出和导入符号信息。它不包含可执行代码，仅用于辅助动态链接器进行符号解析和依赖检查。在加载 `.Wdll` 模块之前，动态链接器可以先读取对应的 `.wli` 文件，了解该模块提供了哪些接口、依赖了哪些外部符号，从而进行预检查和依赖解析。

## 3. 动态链接器工作原理

动态链接器（DynLink）是 WebOS 内核的关键子系统之一，负责在运行时加载 `.Wdll` 动态链接库模块，解析符号引用，并将不同模块链接在一起形成完整的运行时系统。由于 WebAssembly 本身支持动态链接（通过 Emscripten 的 SIDE_MODULE 机制），WebOS 的动态链接器在此基础上进行了封装，提供了统一的加载、符号解析和生命周期管理接口。

### 核心数据结构

```c
/* 动态链接器跟踪的模块信息 */
typedef struct {
    module_handle_t handle;       /* 模块句柄（唯一标识符） */
    char name[64];                /* 模块名称 */
    void* wasm_module;            /* WebAssembly.Module 编译后句柄 */
    void* wasm_instance;          /* WebAssembly.Instance 实例化后句柄 */
    uint32_t ref_count;           /* 引用计数 */
    uint32_t type;                /* 模块类型 (wex/Wdll/wli) */
} dynlink_module_t;

#define MAX_LOADED_MODULES 32     /* 最大同时加载模块数 */
```

动态链接器维护一个包含 32 个槽位的模块表，每个槽位存储一个已加载模块的完整信息。模块句柄是一个递增的 uint32_t 值，用于快速索引和查找。引用计数机制确保模块不会被过早卸载——只有当所有引用该模块的进程都释放了句柄，引用计数降为零时，模块才会被卸载并回收资源。

### 加载与符号解析流程

```
  应用程序调用 SYS_dlopen("wm_service.Wdll")
                    │
                    ▼
  ┌─────────────────────────────────────────┐
  │ 1. 检查模块是否已加载                      │
  │    - 在模块表中搜索名称匹配项               │
  │    - 如果已加载：增加引用计数，返回句柄      │
  └──────────────────┬──────────────────────┘
                     │ 未加载
                     ▼
  ┌─────────────────────────────────────────┐
  │ 2. 通过宿主桥接器获取 WASM 二进制           │
  │    - 宿主运行时通过 Fetch API 获取文件      │
  │    - 读取 module_type 自定义段验证类型      │
  └──────────────────┬──────────────────────┘
                     │
                     ▼
  ┌─────────────────────────────────────────┐
  │ 3. 编译并实例化 WASM 模块                  │
  │    - WebAssembly.compile(binary)          │
  │    - WebAssembly.instantiate(module, imp) │
  │    - 传入 webos.syscall_invoke 和 memory  │
  └──────────────────┬──────────────────────┘
                     │
                     ▼
  ┌─────────────────────────────────────────┐
  │ 4. 注册到模块表                           │
  │    - 分配模块句柄                         │
  │    - 记录名称、类型、实例信息               │
  │    - 设置引用计数为 1                     │
  └──────────────────┬──────────────────────┘
                     │
                     ▼
  ┌─────────────────────────────────────────┐
  │ 5. 返回模块句柄给调用者                    │
  └─────────────────────────────────────────┘
```

当应用程序需要使用某个动态库时，它首先通过 `SYS_dlopen` 系统调用请求内核加载指定名称的模块。动态链接器首先在已加载模块表中查找是否已有同名模块。如果模块已经加载，只需增加引用计数并返回现有句柄，避免重复加载和实例化。如果模块尚未加载，动态链接器通过宿主桥接器（RuntimeBridge）的 `host_syscall_invoke` 接口请求宿主运行时获取 WASM 二进制数据，然后编译并实例化该模块。

符号解析通过 `SYS_dlsym` 系统调用实现。应用程序在获得模块句柄后，可以调用 `dlsym` 获取模块中导出函数的地址。动态链接器根据模块句柄查找到对应的 WASM 实例，然后从实例的导出对象中查找指定名称的函数，返回其包装后的函数指针。这个函数指针可以直接在 C 代码中通过函数指针语法调用。

### 引用计数与卸载

```c
/* 加载模块增加引用计数 */
module_handle_t dynlink_load(const char* name, int32_t name_len) {
    // 搜索已加载模块
    dynlink_module_t* existing = dynlink_get_by_name(name);
    if (existing) {
        existing->ref_count++;
        return existing->handle;
    }
    // ... 新加载逻辑 ...
}

/* 卸载模块减少引用计数 */
int32_t dynlink_unload(module_handle_t handle) {
    dynlink_module_t* mod = dynlink_get_info(handle);
    if (!mod) return -2;  // ERR_NOENT
    mod->ref_count--;
    if (mod->ref_count == 0) {
        // 清理资源，从模块表中移除
    }
    return 0;
}
```

引用计数机制确保了共享库的安全性。当多个进程同时依赖同一个 `.Wdll` 模块时，模块只会被加载一次，但引用计数会递增反映实际使用情况。只有当所有依赖方都调用了 `dynlink_unload()` 释放句柄后，引用计数归零，模块才会被真正卸载。这种设计既节省了内存资源，又避免了悬挂引用导致的运行时错误。

## 4. 系统调用接口概述

系统调用（Syscall）是用户空间模块与内核交互的唯一标准接口。在 WebOS 中，系统调用通过 WASM 的导入函数机制实现：用户空间模块导入 `webos.syscall_invoke` 函数，通过传递系统调用号和最多 5 个整数参数来请求内核服务。这种设计将所有特权操作都路由到内核，实现了清晰的权限隔离和安全检查。

### 调用约定

```c
/* 用户空间系统调用桥接函数 */
long kernel_syscall(long n, long a1, long a2, long a3);

/* 便捷封装 */
static inline long syscall0(long n) {
    extern long kernel_syscall(long, long, long, long);
    return kernel_syscall(n, 0, 0, 0);
}

static inline long syscall1(long n, long a1) {
    extern long kernel_syscall(long, long, long, long);
    return kernel_syscall(n, a1, 0, 0);
}

static inline long syscall2(long n, long a1, long a2) {
    extern long kernel_syscall(long, long, long, long);
    return kernel_syscall(n, a1, a2, 0);
}

static inline long syscall3(long n, long a1, long a2, long a3) {
    extern long kernel_syscall(long, long, long, long);
    return kernel_syscall(n, a1, a2, a3);
}
```

### 系统调用编号分配

WebOS 采用分区间编号方案，每个子系统占据一个 256 个编号的区间，高 8 位标识子系统类别，低 8 位标识具体调用。这种设计使得编号具有自描述性，便于开发者快速识别系统调用所属类别，也为未来扩展预留了充足的空间。

| 区间 | 起始编号 | 子系统 | 描述 |
|------|---------|--------|------|
| FS | 0x0100 | 文件系统 | 文件读写、目录操作 |
| PROCESS | 0x0200 | 进程管理 | 进程创建、终止、查询 |
| MEMORY | 0x0300 | 内存管理 | 内存分配、释放、映射 |
| TIME | 0x0400 | 时间服务 | 获取系统时间 |
| IO | 0x0500 | 输入输出 | 设备 I/O 操作 |
| NET | 0x0600 | 网络 | 网络请求 |
| DEBUG | 0x0700 | 调试 | 日志输出、调试信息 |
| IPC | 0x0800 | 进程间通信 | 消息发送、接收 |
| NOTIFY | 0x0900 | 通知 | 系统通知与事件 |
| DYNLINK | 0x0A00 | 动态链接 | 模块加载、符号解析 |

### 调用流程

```
  用户空间模块                    内核                          宿主运行时
  ════════════                  ════                          ════════

  syscall1(SYS_exit, code)
       │
       ▼
  kernel_syscall(0x0200, code, 0, 0)
       │  ← WASM 导入函数调用
       ▼
  syscall_dispatch(0x0200, code, 0, 0, 0, 0)
       │
       ├─ 查找分发表：handlers[0x0200]
       │
       ▼
  process_exit(code)  ← 对应处理器函数
       │
       ▼
  host_syscall_invoke(...)  ← 需要宿主协助时
       │
       ▼
  RuntimeBridge.handleSyscall(...)
       │
       ▼
  浏览器 API 调用
```

系统调用的完整流程涉及三个层面：用户空间的系统调用封装函数、内核的系统调用分发表、以及宿主运行时的具体实现。当用户空间代码调用 `syscall_exit(code)` 时，内联封装函数将其转换为 `kernel_syscall(SYS_exit, code, 0, 0)` 调用。这个函数是 WASM 导入函数，由内核模块导出。内核的 `syscall_dispatch()` 函数接收调用请求，根据系统调用号在分发表中查找对应的处理函数，然后调用处理器执行具体逻辑。如果处理器需要访问浏览器 API（如文件操作需要 IndexedDB），则通过 `host_syscall_invoke()` 转发到宿主运行时的 `RuntimeBridge`。

### 错误码

所有系统调用返回 `int32_t` 值，0 表示成功，负值表示错误：

| 错误码 | 常量 | 含义 |
|-------|------|------|
| 0 | `ERR_OK` | 成功 |
| -1 | `ERR_INVAL` | 无效参数 |
| -2 | `ERR_NOENT` | 条目不存在 |
| -3 | `ERR_NOMEM` | 内存不足 |
| -4 | `ERR_BUSY` | 资源忙 |
| -5 | `ERR_PERM` | 权限不足 |
| -6 | `ERR_FAULT` | 地址错误 |
| -7 | `ERR_EXIST` | 已存在 |
| -8 | `ERR_NOSYS` | 未实现 |

## 5. IPC 通信机制

进程间通信（IPC）是 WebOS 中不同模块之间协作的核心机制。由于 WebAssembly 运行在单一地址空间中，传统的基于共享内存的 IPC 模型需要额外的同步机制。WebOS 采用了基于消息传递的 IPC 模型，每个进程拥有独立的消息队列，通过内核的 IPC 子系统进行消息路由和传递。这种设计保证了进程间的隔离性，同时提供了同步和异步两种通信模式。

### 消息格式

```c
/* IPC 消息类型 */
#define IPC_MSG_SYSCALL   0x01   /* 系统调用请求 */
#define IPC_MSG_SIGNAL    0x02   /* 信号通知 */
#define IPC_MSG_DATA      0x03   /* 数据传输 */
#define IPC_MSG_REQUEST   0x04   /* 服务请求 */
#define IPC_MSG_RESPONSE  0x05   /* 服务响应 */
#define IPC_MSG_EVENT     0x06   /* 系统事件 */

#define IPC_MAX_MSG_SIZE  4096   /* 单条消息最大字节数 */
#define IPC_MAX_PENDING   32     /* 每进程最大待处理消息数 */

/* IPC 消息结构 */
typedef struct {
    pid_t sender;                        /* 发送者进程 ID */
    pid_t receiver;                      /* 接收者进程 ID */
    uint32_t type;                       /* 消息类型 */
    uint32_t length;                     /* 数据长度 */
    uint8_t data[IPC_MAX_MSG_SIZE];      /* 消息数据 */
} ipc_msg_t;

/* IPC 通道 */
typedef struct {
    pid_t owner;                         /* 通道拥有者 */
    uint32_t msg_count;                  /* 当前消息数 */
    uint32_t capacity;                   /* 通道容量 */
    ipc_msg_t messages[IPC_MAX_PENDING]; /* 消息缓冲区 */
} ipc_channel_t;
```

### 通信模式

WebOS 支持三种 IPC 通信模式，分别适用于不同的应用场景：

**异步发送模式**：调用 `ipc_send(target, type, data, len)` 将消息放入目标进程的消息队列后立即返回，不等待对方处理。适用于事件通知、日志上报等不需要即时响应的场景。发送方不会被阻塞，可以继续执行后续逻辑。如果目标进程的消息队列已满（达到 `IPC_MAX_PENDING=32` 条），则发送操作返回 `ERR_BUSY` 错误。

**非阻塞接收模式**：调用 `ipc_recv_nonblock(msg)` 尝试从当前进程的消息队列中取出一条消息。如果队列中有消息，立即返回并填充 `msg` 结构；如果队列为空，返回 `ERR_BUSY` 而不阻塞。这种模式适合在应用的主循环中轮询消息，与 Emscripten 的事件驱动模型配合使用。

**同步调用模式**：调用 `ipc_call(target, type, data, len, resp, resp_len)` 发送请求后阻塞等待目标进程的响应。目标进程处理完请求后调用 `ipc_reply(caller, data, len)` 发送响应。这种模式适用于需要即时结果的远程过程调用（RPC）场景，如应用程序请求文件系统服务打开文件。内核在同步调用期间将调用方进程的状态设置为 `PROC_STATE_BLOCKED`，直到收到响应或超时才恢复为 `PROC_STATE_READY`。

### IPC 通信流程图

```
  进程 A (应用)                   内核 IPC                    进程 B (服务)
  ═════════════                  ═══════                    ════════════

  ipc_call(pid_B, REQUEST,
           &req, sizeof(req),
           &resp, &resp_len)
       │
       ▼
                                 ┌─────────────────┐
                                 │ 1. 分配消息缓冲   │
                                 │ 2. 复制请求数据   │
                                 │ 3. 放入 B 的队列  │
                                 │ 4. 阻塞进程 A    │
                                 └────────┬────────┘
                                          │
                                          ▼
                                                              ipc_recv(&msg)
                                                              处理请求...
                                                              ipc_reply(pid_A,
                                                              &resp, len)
                                                                  │
                                                                  ▼
                                 ┌─────────────────┐
                                 │ 5. 复制响应数据   │
                                 │ 6. 唤醒进程 A    │
                                 └────────┬────────┘
                                          │
                                          ▼
  ipc_call() 返回
  响应数据在 resp 中
```

### 服务发现与请求路由

应用程序通常不需要直接知道服务进程的 PID，而是通过服务名称进行请求。IPC 子系统提供了基于服务名称的路由机制：服务进程在启动时通过 `ipc_register_service(name)` 注册自己的服务名称，内核维护一个服务名称到 PID 的映射表。应用程序调用 `ipc_call_service("wm_service", ...)` 时，内核自动查找对应服务的 PID 并路由消息。这种间接寻址方式解耦了服务提供者和服务消费者，使得服务进程可以重启而不影响客户端。

## 6. 进程调度机制

WebOS 的进程调度器运行在 WebAssembly 的单线程环境中，这意味着它无法使用传统的抢占式调度。WebOS 采用了协作式调度（Cooperative Scheduling）模型，每个进程自愿让出 CPU 时间，调度器在进程让出时选择下一个运行的进程。这种模型与浏览器的事件循环天然契合，Emscripten 的 `emscripten_set_interval()` 和 `emscripten_set_main_loop()` 机制为实现协作式调度提供了基础。

### 进程状态机

```
                          process_spawn()
                               │
                               ▼
                    ┌──────────────────┐
                    │   PROC_READY     │ ◀── scheduler_unblock()
                    │   (就绪状态)      │
                    └────────┬─────────┘
                             │ scheduler 选中
                             ▼
                    ┌──────────────────┐
              ┌────▶│  PROC_RUNNING    │────┐
              │     │  (运行状态)       │    │
              │     └──────────────────┘    │
              │                             │
              │  scheduler_unblock()        │ scheduler_yield()
              │                             │ scheduler_block()
              │                             ▼
              │                    ┌──────────────────┐
              │                    │  PROC_BLOCKED    │
              │                    │  (阻塞状态)       │
              │                    │  等待 IPC 响应    │
              │                    │  等待 I/O 完成    │
              │                    └──────────────────┘
              │
              │                             │ process_exit()
              │                             ▼
              │                    ┌──────────────────┐
              │                    │  PROC_ZOMBIE     │
              │                    │  (僵尸状态)       │
              │                    │  等待父进程回收    │
              │                    └──────────────────┘
              │
              └──── process_wait() 回收后释放
```

### 进程控制块

```c
#define MAX_PROCESSES 64           /* 最大进程数 */

#define PROC_PRIORITY_LOW      0   /* 低优先级 */
#define PROC_PRIORITY_NORMAL   1   /* 普通优先级 */
#define PROC_PRIORITY_HIGH     2   /* 高优先级 */
#define PROC_PRIORITY_REALTIME 3   /* 实时优先级 */

typedef struct {
    pid_t pid;                     /* 进程 ID */
    uint32_t state;                /* 进程状态 */
    uint32_t priority;             /* 优先级 */
    void* module_handle;           /* WASM 模块句柄 */
    void* stack_ptr;               /* 当前栈指针 */
    void* entry_point;             /* 程序入口地址 */
    uint32_t flags;                /* 进程标志位 */
    int32_t exit_code;             /* 退出码 */
} process_t;
```

每个进程在内核中都有一个进程控制块（PCB），记录了进程的全部运行时状态。系统最多支持 64 个并发进程。进程 ID 从 1 开始递增分配，PID 0 保留给内核自身。进程的优先级分为四个级别：低优先级适用于后台任务（如日志收集），普通优先级是大多数应用的默认级别，高优先级适用于需要快速响应的服务（如窗口管理器），实时优先级保留给对时延极度敏感的任务。

### 调度算法

WebOS 支持两种调度算法，可通过 `scheduler_set_config()` 动态切换：

**轮转调度（Round Robin）**：所有就绪状态的进程按 FIFO 顺序排入就绪队列。调度器从队列头部取出一个进程执行，当进程调用 `scheduler_yield()` 主动让出 CPU、或者时间片耗尽时，调度器将当前进程放回队列尾部，从队列头部取出下一个进程执行。轮转调度保证了所有进程获得公平的 CPU 时间，适合交互式场景。

**优先级调度（Priority Scheduling）**：进程按优先级分组，高优先级组中的进程优先执行。同一优先级组内采用轮转调度。只有当高优先级组中没有就绪进程时，低优先级组的进程才会获得执行机会。这种算法适用于需要区分服务等级的场景，如确保窗口管理器始终能及时响应用户操作。

```c
/* 调度器配置 */
typedef struct {
    uint32_t algorithm;         /* SCHED_ROUND_ROBIN 或 SCHED_PRIORITY */
    uint32_t time_slice_ms;     /* 时间片长度（毫秒） */
} sched_config_t;
```

### 协作式调度与事件循环

在 WebAssembly 单线程环境中，调度器不能中断正在运行的进程。因此，进程必须主动调用 `scheduler_yield()` 让出 CPU，否则其他进程将无法执行。WebOS 的应用程序通常采用以下模式配合调度器工作：

```c
/* 应用主循环 */
void app_tick(void* arg) {
    /* 1. 处理输入事件 */
    wm_event_t event;
    while (wm_poll_event(win, &event) > 0) {
        handle_event(&event);
    }

    /* 2. 更新应用状态 */
    update_state();

    /* 3. 渲染画面 */
    render_frame();

    /* 4. 让出 CPU（emscripten_set_interval 自动实现） */
    /* emscripten 的定时回调机制天然实现了协作式让出 */
}

int main(void) {
    /* 创建窗口等初始化工作 */
    int win = wm_create_window("App", 100, 100, 400, 300);

    /* 注册定时回调，约 60fps */
    emscripten_set_interval(app_tick, 16, NULL);

    return 0;
}
```

Emscripten 的 `emscripten_set_interval()` 将应用逻辑注册为浏览器事件循环的回调函数。每次回调执行完毕后，控制权自然返回浏览器事件循环，调度器可以在此间隙切换到其他进程。这种模式将协作式调度与浏览器事件循环完美融合，既保证了系统的响应性，又简化了应用开发。

## 7. 内存管理架构

WebOS 的内存管理建立在 WebAssembly 线性内存之上。所有 WASM 模块共享同一个 `WebAssembly.Memory` 实例，由宿主运行时的 RuntimeBridge 创建和管理。这种单地址空间模型简化了模块间的数据交换，但要求内核对内存进行严格的管理和隔离，防止不同进程的内存区域相互干扰。

### 分配与回收

内核的内存管理器采用自由链表（Free List）分配算法。系统初始化时，整个可用内存空间作为一个大的空闲区域被加入自由链表。当进程请求内存时（通过 `SYS_malloc` 系统调用），内存管理器在自由链表中查找足够大的空闲区域，将其分割为已分配部分和剩余空闲部分，并将剩余部分重新加入自由链表。分配的内存区域记录了所有者进程 ID，当进程退出时，内核自动回收该进程拥有的所有内存区域。

### 共享内存

共享内存区域是 IPC 通信的重要补充。当进程之间需要传输大量数据（如位图像素数据）时，基于消息的 IPC 效率较低，因为消息数据需要在发送方和接收方之间复制。共享内存允许两个或多个进程映射同一块内存区域，通过直接读写共享区域实现零拷贝数据交换。内核通过 `memory_map_shared()` 创建共享内存区域，记录参与进程的 PID 列表。当进程通过 IPC 交换共享内存的地址和长度信息后，双方即可直接读写共享区域。

共享内存的同步通过 IPC 消息实现：发送方在数据写入完成后发送一条通知消息，接收方收到通知后读取共享内存数据。这种约定式的同步机制在单线程的 WebAssembly 环境中是安全可靠的，因为不存在并发访问的问题——同一时刻只有一个进程在运行。

### 内存增长策略

WebAssembly 的线性内存支持动态增长，但不能缩小。宿主运行时创建 `WebAssembly.Memory` 时设置了 `initial: 256`（16MB 初始大小）和 `maximum: 1024`（64MB 最大大小），并启用了 `ALLOW_MEMORY_GROWTH`。当内核的内存管理器发现自由链表中没有足够大的连续区域满足分配请求时，会通过 `memory.grow()` WASM 指令增长线性内存，每次增长一个或多个 64KB 页面。增长后的新空间被加入自由链表，然后重新尝试分配。
