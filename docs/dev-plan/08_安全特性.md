# 安全特性

WebOS 的安全模型来自一个根本性的架构决策——**整个系统渲染在一个画布上，没有 DOM**。

---

## 无 DOM = 无 XSS

传统 Web 应用的安全噩梦是 XSS（跨站脚本攻击）。攻击者注入的 JavaScript 可以:
- 读取页面中的敏感信息（cookie, token）
- 修改 DOM 窃取用户输入
- 发起恶意请求

WebOS 完全不同。浏览器的 F12 开发者工具看到的是:

```html
<body>
  <canvas id="webos-canvas"></canvas>
</body>
```

就这样。没有 `<input>`, 没有 `<div>`, 没有任何可被注入的 DOM 节点。

### 攻击面对比

| 攻击向量 | 传统 Web 应用 | WebOS |
|----------|--------------|-------|
| DOM 注入 | ✅ 可以注入任意 HTML/JS | ❌ 没有 DOM |
| 表单窃取 | ✅ 可以读 input.value | ❌ 输入框是画布上的像素 |
| Cookie 窃取 | ✅ document.cookie | ❌ 无 cookie 机制 |
| 事件劫持 | ✅ addEventListener 拦截 | ❌ 事件由宿主统一分发 |
| ifram 嵌入 | ✅ 可以嵌入恶意页面 | ❌ 不使用 ifram |
| CSS 注入 | ✅ 可以覆盖样式 | ❌ 没有 CSS |
| localStorage 读取 | ✅ 同源可读 | ❌ 使用 IndexedDB 独立存储 |

---

## 自主输入处理

所有用户输入——键盘、鼠标、触摸——都由 TypeScript 宿主直接捕获，然后通过 `RuntimeBridge` 分发给 WASM 模块:

```
浏览器事件 → canvas.addEventListener() → RuntimeBridge → WASM IPC → 目标进程
```

没有浏览器原生的 `<input>` 元素，所以:
- 光标闪烁是画出来的
- 文字输入是逐键处理的
- 按键映射是自定义的
- 鼠标位置是像素坐标

这意味着即使攻击者在某个应用中执行了代码，他也无法:
- 创建隐藏的输入框
- 拦截其他应用的键盘输入
- 模拟用户点击浏览器的原生控件

---

## 进程隔离

每个应用运行在独立的 WASM 模块中，通过 IPC 通信:

- 进程无法直接访问其他进程的内存
- 所有跨进程操作必须通过内核系统调用
- 文件系统访问受权限控制
- 设备驱动访问通过服务层代理

---

## 存储安全

WebFS 使用 IndexedDB 作为后端存储:

- 每个文件有 inode 权限位
- 应用只能访问其权限范围内的路径
- 没有 `document.cookie` 可以窃取
- 没有 `localStorage` 可以读取

---

## WebGPU 渲染安全

所有画面输出通过 WebGPU 渲染:

- 渲染管线完全由宿主控制
- 应用无法注入着色器代码到其他窗口
- 窗口内容隔离在各自的 RenderTarget 中
- 最终合成由 Compositor 统一完成

### 合成器架构

```
App A → RenderTarget A ─┐
App B → RenderTarget B ─┼→ Compositor → 主画布
App C → RenderTarget C ─┘
```

应用只能写入自己的 RenderTarget，无法影响其他窗口的渲染。
